# discord-mcp-plus v0.5 — Init CLI, Dry-Run Enforcement, Confirmation Tokens

**Date:** 2026-04-14
**Status:** Approved

---

## Goal

Three operator-safety features that make the server safer to deploy and hand to AI agents:

1. **`init` CLI** — `npx discord-mcp-plus init` prints a ready-to-use `.env` template to stdout
2. **Dry-run enforcement** — `DISCORD_MCP_DRY_RUN=true` actually intercepts write/destructive tool calls and returns a simulated response instead of executing
3. **Confirmation tokens** — `DISCORD_MCP_CONFIRM_TOKEN=<secret>` requires destructive tool calls to include a matching `confirmToken` arg, preventing accidental/unauthorized destructive actions

## Feature Specifications

### 1. `init` CLI

**Invocation:** `npx discord-mcp-plus init` (or `node dist/index.js init`)

**Behavior:** Prints a `.env` template to stdout and exits with code 0. No file is written — the user pipes it: `npx discord-mcp-plus init > .env`.

**Template contents** (all vars, with comments):

```
# discord-mcp-plus configuration
# Generated by: npx discord-mcp-plus init

# Required
DISCORD_TOKEN=your-bot-token-here

# Optional — restricts all guild-scoped tools to this guild
# DISCORD_GUILD_ID=

# Safety — set to true to hide write/destructive tools from the AI
# DISCORD_MCP_READONLY=false

# Safety — comma-separated tool names to allow (all others hidden)
# DISCORD_MCP_TOOLS=

# Safety — comma-separated tool names to deny
# DISCORD_MCP_TOOLS_DENY=

# Safety — comma-separated channel IDs where write operations are permitted
# DISCORD_MCP_CHANNELS=

# Safety — intercept write/destructive calls and return a simulated response
# DISCORD_MCP_DRY_RUN=false

# Safety — require this secret token as confirmToken arg on all destructive tool calls
# DISCORD_MCP_CONFIRM_TOKEN=

# Audit — 'stderr' (default), 'off', or a file path
# DISCORD_MCP_AUDIT_LOG=stderr
```

**Implementation:** `src/init.ts` exports `printEnvTemplate(): void` which calls `process.stdout.write(template)`. `src/index.ts` checks `process.argv[2] === 'init'`, calls `printEnvTemplate()`, and exits before login.

### 2. Dry-Run Enforcement

**Config field:** `config.dryRun: boolean` (already parsed, currently ignored)

**Enforcement point:** `ToolRegistry.call()` — after visibility and channel checks, before calling the handler.

**Logic:**
```
if config.dryRun AND tool.kind is 'write' or 'destructive':
  return { dryRun: true, tool: name, args, note: 'Dry run mode — action was not executed' }
```

Read tools always execute regardless of dryRun (they have no side effects).

**Startup warning:** When `config.dryRun === true`, `main()` writes to stderr:
```
[discord-mcp-plus] DRY RUN MODE — write and destructive tools will not execute
```

**Audit behavior:** Dry-run interceptions are NOT audited (no real action occurred). This is consistent with the existing policy of only auditing successful writes/destructives.

### 3. Confirmation Tokens

**Config field:** `config.confirmationToken: string | false` — parsed from `DISCORD_MCP_CONFIRM_TOKEN`. `false` means feature is disabled.

**Enforcement point:** `ToolRegistry.call()` — after dry-run check, only for `destructive` tools.

**Logic:**
```
if config.confirmationToken AND tool.kind is 'destructive':
  if args.confirmToken !== config.confirmationToken:
    throw new Error('Destructive action requires a valid confirmToken')
```

**Schema injection:** `ToolRegistry.listVisible()` returns modified copies of destructive tool schemas when `confirmationToken` is configured — injects `confirmToken` as a required field so the MCP client (and AI) know to supply it:

```typescript
// Injected into destructive tool inputSchema when confirmationToken is set:
confirmToken: {
  type: 'string',
  description: 'Confirmation token required for destructive actions (set via DISCORD_MCP_CONFIRM_TOKEN)'
}
```

The field is added to `properties` but NOT to `required` — this is intentional. The AI sees the field and its description, understands it must be supplied, but the hard enforcement is in the registry. Adding it to `required` would cause MCP clients that validate schemas to reject calls that don't include it before the server even processes them — which would produce a less informative error.

**Startup warning:** When `confirmationToken` is set, `main()` writes to stderr:
```
[discord-mcp-plus] CONFIRM TOKEN active — destructive tools require confirmToken arg
```

**Audit behavior:** Failed token checks ARE audited (same as other call failures — all failures audit regardless of kind).

## Testing Strategy

**`tests/registry.test.ts`** (add 6 tests):
1. `dryRun=true` intercepts write tool — returns `{ dryRun: true }` shape
2. `dryRun=true` intercepts destructive tool — returns `{ dryRun: true }` shape
3. `dryRun=true` does NOT intercept read tool — handler executes normally
4. `confirmationToken` set, correct token passes — handler executes
5. `confirmationToken` set, wrong token throws with descriptive message
6. `confirmationToken` set, missing token throws with descriptive message
7. `listVisible()` injects `confirmToken` into destructive tool schema when token configured
8. `listVisible()` does NOT inject `confirmToken` when no token configured

**`tests/init.test.ts`** (2 tests):
1. `printEnvTemplate()` output contains all required variable names
2. Output contains `DISCORD_TOKEN` as the only uncommented required var
