# discord-mcp-plus v0.1 — Foundation Design Spec

**Date:** 2026-04-14
**Status:** Approved
**Milestone:** v0.1 — scaffolding, config system, tool registry with safety gating, server-info tools

---

## Overview

`discord-mcp-plus` is a production-grade open-source MCP server for Discord. It gives AI assistants (Claude Desktop, Claude Code, Cursor, Windsurf) full read/write access to a Discord server, with a safety layer that makes the safe path the easy path.

v0.1 delivers the foundation everything else builds on: a validated config system, a tool registry that enforces all safety policy, and two initial tools (`get_server_info`, `list_bot_permissions`). No tools from the personal `discord_mcp` project are ported yet — that happens in v0.2.

## Config System (`src/config.ts`)

### Environment Variables

| Variable | Type | Default | Description |
|----------|------|---------|-------------|
| `DISCORD_TOKEN` | string | — | Bot token. **Required.** Throws on missing. |
| `DISCORD_GUILD_ID` | string | — | Default guild ID. When set, `guildId` param becomes optional on all guild-scoped tools. |
| `DISCORD_MCP_READONLY` | boolean | `false` | When `true`, all `write` and `destructive` tools are hidden from ListTools. |
| `DISCORD_MCP_TOOLS` | string | — | Comma-separated allowlist of tool names. Only listed tools appear. |
| `DISCORD_MCP_TOOLS_DENY` | string | — | Comma-separated denylist. Takes precedence over allowlist. |
| `DISCORD_MCP_CHANNELS` | string | — | Comma-separated channel ID allowlist. Write/destructive tool calls targeting unlisted channels are rejected at call time. |
| `DISCORD_MCP_DRY_RUN` | boolean | `false` | Reserved for v0.5. Parsed and stored but not enforced in v0.1. |
| `DISCORD_MCP_AUDIT_LOG` | string | `stderr` | Path to write NDJSON audit log. `off` disables. Defaults to `stderr` (not `stdout` — stdout is reserved for the MCP stdio transport). |
| `TRANSPORT` | `'stdio'` | `'stdio'` | Only `stdio` supported in v0.1. Parsed for forward-compatibility. |

### Config Interface

```typescript
interface Config {
  token: string
  guildId: string | undefined
  readonly: boolean
  toolsAllow: Set<string> | null   // null = no allowlist (all tools visible)
  toolsDeny: Set<string>           // empty = no denylist
  channelsAllow: Set<string> | null // null = no channel restriction
  dryRun: boolean
  auditLog: string | false         // 'stderr', file path, or false (disabled)
  transport: 'stdio'
}
```

### Parsing Rules

- Missing `DISCORD_TOKEN` → throw with message: `"DISCORD_TOKEN is required"`
- Comma-separated lists: split on `,`, trim whitespace, filter empty strings, convert to `Set`
- Empty string for list vars → treated as unset (no restriction)
- Boolean vars: `'true'` (case-insensitive) → `true`, anything else → `false`
- `DISCORD_MCP_AUDIT_LOG=off` → `auditLog: false`
- `DISCORD_MCP_AUDIT_LOG` unset → `auditLog: 'stderr'` (stdout is reserved for MCP stdio transport)

## Tool Registry (`src/registry.ts`)

The registry is the single place where all safety policy is enforced. Tools register with metadata; the registry decides what's visible and what's executable.

### Tool Definition

```typescript
type ToolKind = 'read' | 'write' | 'destructive'

interface ToolDef {
  name: string
  description: string
  inputSchema: Record<string, unknown>   // JSON Schema object
  kind: ToolKind
  handler: (args: Record<string, unknown>, config: Config, client: Client) => Promise<unknown>
}
```

### Safety Policy (enforced by registry)

**At ListTools time (what the agent sees):**
1. `readonly: true` → hide all `write` and `destructive` tools
2. `toolsAllow` is set → only include tools whose name is in the set
3. `toolsDeny` is non-empty → exclude tools whose name is in the set
4. Deny takes precedence over allow

**At CallTool time (when the agent calls a tool):**
1. Re-verify the tool passes ListTools policy (defense in depth)
2. For `write` or `destructive` tools: if `channelsAllow` is set, extract `channelId` from args — if present and not in `channelsAllow`, reject with error: `"Channel <id> is not in the allowed channel list"`
3. After successful call: if tool is `write` or `destructive`, write audit log entry

### Audit Log Format

NDJSON, one entry per line:

```json
{"ts":"2026-04-14T20:00:00.000Z","tool":"send_message","kind":"write","args":{"channelId":"123","content":"hello"},"durationMs":142,"result":"ok"}
{"ts":"2026-04-14T20:00:01.000Z","tool":"delete_message","kind":"destructive","args":{"channelId":"123","messageId":"456"},"durationMs":89,"result":"error","error":"Missing Permissions"}
```

Written to `auditLog` path (or stdout if `'stdout'`). File opened in append mode. If write fails, log warning to stderr and continue — never crash on audit failure.

## Discord Client (`src/discord-client.ts`)

Same pattern as `discord_mcp` with fixes applied:
- `loginClient(token)` registers `ready` listener **before** `client.login()`
- Re-entrant guard: throws if called twice
- `getClient()` throws if not initialized
- All logging to `console.error` (stdout reserved for MCP transport)

Intents: `Guilds`, `GuildMembers`, `GuildMessages`, `MessageContent`, `DirectMessages`
Partials: `Channel`, `Message`

---

## MCP Server (`src/server.ts`)

Creates the MCP `Server` instance, wires `ListToolsRequestSchema` and `CallToolRequestSchema` handlers to the registry, connects `StdioServerTransport`. Error responses use `isError: true` and log to stderr.

---

## Entry Point (`src/index.ts`)

Bin entry with `#!/usr/bin/env node` shebang (injected by tsup). Sequence:

1. Load `.env` via `dotenv/config`
2. Call `loadConfig()` — throws on missing token
3. Call `loginClient(config.token)`
4. Create registry, register all tools
5. Call `startServer(registry, config)`

On any startup failure: `console.error` + `process.exit(1)`.

## v0.1 Tools (`src/tools/server-info.ts`)

### `get_server_info` (`read`)

Returns guild metadata. Uses `config.guildId` as default if no `guildId` arg provided.

**Input:** `{ guildId?: string }`
**Output:**
```json
{
  "id": "123",
  "name": "My Server",
  "description": "...",
  "memberCount": 42,
  "channelCount": 15,
  "createdAt": "2020-01-01T00:00:00.000Z",
  "premiumTier": 0,
  "features": ["COMMUNITY"]
}
```

### `list_bot_permissions` (`read`)

Returns what permissions the bot currently has in the guild (or a specific channel). Useful for agents to self-check before attempting writes.

**Input:** `{ guildId?: string, channelId?: string }`
**Output:**
```json
{
  "scope": "guild",
  "permissions": ["SendMessages", "ReadMessageHistory", "ManageChannels"],
  "missing": ["Administrator", "KickMembers"]
}
```

When `channelId` is provided, returns effective permissions in that channel (respecting overrides).

## Testing

### `tests/config.test.ts`

- Missing `DISCORD_TOKEN` throws
- All defaults applied correctly when only token set
- `DISCORD_MCP_READONLY=true` parses to `true`, other values parse to `false`
- Comma list `"foo,bar, baz"` parses to `Set { 'foo', 'bar', 'baz' }`
- Empty string for list var → `null` (no restriction)
- `DISCORD_MCP_AUDIT_LOG=off` → `false`
- `DISCORD_MCP_AUDIT_LOG` unset → `'stdout'`

### `tests/registry.test.ts`

Uses stub `ToolDef` objects (no real discord.js). Tests:
- `readonly: true` → write + destructive tools absent from ListTools, read tools present
- `toolsAllow: Set{'foo'}` → only `foo` appears in ListTools
- `toolsDeny: Set{'foo'}` → `foo` absent from ListTools, others present
- Deny + allow together: deny wins
- `channelsAllow` rejects write call with unlisted channelId
- `channelsAllow` allows write call with listed channelId
- Audit log entry written after successful write tool call
- Audit log entry written (with `result: 'error'`) after failed write tool call
