## docs/bot-setup.md

Step-by-step manual guide for creating a Discord bot. Referenced from README quick-start as the manual alternative to `npx discord-mcp-plus init`.

Content:
1. **Create the application** — discord.com/developers/applications → New Application → name it
2. **Create the bot** — Application → Bot → Add Bot → copy the token → store it as `DISCORD_TOKEN`
3. **Enable privileged intents** — Bot settings → enable "Server Members Intent" and "Message Content Intent" (both required)
4. **Generate an invite URL** — OAuth2 → URL Generator → scope: `bot` → permissions: `Read Messages/View Channels`, `Send Messages`, `Read Message History`, `Add Reactions`, `Manage Messages`, `Manage Channels`, `Kick Members`, `Ban Members`, `Manage Roles`, `Attach Files` → copy the URL → open in browser → invite to your server
5. **Create .env** — copy `.env.example` to `.env`, fill in `DISCORD_TOKEN` and optionally `DISCORD_GUILD_ID`

## CONTRIBUTING.md

Contributor guidelines. Content:

**Running locally:**
- `npm install`
- `cp .env.example .env` (fill in a real bot token for integration testing)
- `npm run dev` (stdio transport, connects to Discord)
- `npm test` (unit tests, no Discord connection needed)
- `npm run typecheck`

**Adding a new tool:**
1. Add the `ToolDef` to the appropriate file in `src/tools/` (or create a new file if it's a new domain)
2. Register it in `src/index.ts`
3. Write at least one Vitest test in `tests/`
4. If the tool is `kind: 'destructive'`, it must honor `config.dryRun` and support `confirmToken` via `withConfirmTokenSchema()` in the registry
5. If the tool is a read that returns potentially large lists, support cursor-based pagination

**ToolDef interface:**
```typescript
interface ToolDef {
  name: string          // snake_case
  description: string   // shown to the AI — be specific, include an example
  inputSchema: object   // JSON Schema for args
  kind: 'read' | 'write' | 'destructive'
  handler(args, config, client): Promise<unknown>
}
```

**Test requirements:**
- Every new tool needs at least one test covering the happy path
- Destructive tools need a test for the dry-run path
- Tests mock the Discord client via `vi.fn()` — no real Discord connection
- Run `npm test` before opening a PR

**PR guidelines:**
- One feature or fix per PR
- Conventional commit messages (`feat:`, `fix:`, `ci:`, `docs:`, `chore:`)
- Tests must pass: `npm test && npm run typecheck`
- No new dependencies without discussion
