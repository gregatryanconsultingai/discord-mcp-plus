## Tool Specifications

### `search_messages` (read)

Searches messages in a guild using Discord's native REST search endpoint.

**Inputs:**
- `guildId` â€” string, optional. Falls back to `config.guildId`. Throws if neither is available.
- `content` â€” string, optional. Full-text search term.
- `channelId` â€” string, optional. Restrict results to a specific channel.
- `authorId` â€” string, optional. Restrict results to a specific author.
- `limit` â€” number, optional. Default 25, max 25 (Discord API cap).

**Implementation:** Calls `client.rest.get('/guilds/:guildId/messages/search', { query: { content, channel_id, author_id, limit } })`. Discord returns `{ messages: Message[][], total_results: number }` where each inner array is a search hit plus surrounding context messages. We take `result.messages[i][0]` (the hit) and discard context.

**Returns:**
```typescript
{
  totalResults: number,
  messages: Array<{
    id: string,
    channelId: string,
    author: { id: string, username: string },
    content: string,
    timestamp: string, // ISO 8601
  }>
}
```

**Errors:** Throws if no guildId available. Propagates REST errors from Discord (e.g. 403 Missing Permissions, 404 Unknown Guild).

### `get_messages` (read, enhanced)

Existing tool. Adds two optional pagination params, fully backward-compatible.

**New inputs (both optional):**
- `before` â€” string snowflake. Fetch messages sent before this message ID.
- `after` â€” string snowflake. Fetch messages sent after this message ID.

**Implementation:** Passed directly to `channel.messages.fetch({ limit, before, after })`. discord.js handles them natively. No behavior change when omitted.

**Typical pagination loop:** Call with no cursor to get the first page. The oldest message ID in the response becomes the `before` cursor for the next page.

### `wait_for_message` (read)

Blocks until a message matching all specified filters arrives in the target channel, or the timeout expires.

**Inputs:**
- `channelId` â€” string, required. Channel to listen in.
- `userId` â€” string, optional. Only match messages from this user.
- `pattern` â€” string, optional. Regular expression. Only match messages whose content matches. Compiled at registration time â€” throws immediately if the pattern is invalid regex.
- `timeout` â€” number, optional. Seconds to wait before giving up. Default 30, max 300.

**Returns on match:**
```typescript
{
  timedOut: false,
  message: {
    id: string,
    channelId: string,
    author: { id: string, username: string },
    content: string,
    timestamp: string, // ISO 8601
  }
}
```

**Returns on timeout:**
```typescript
{ timedOut: true, message: null }
```

## Event Dispatcher Architecture

`src/tools/events.ts` owns a module-level `Map<string, Waiter>` where each key is a `crypto.randomUUID()` assigned at registration time.

```
Waiter = {
  resolve: (result: WaitResult) => void
  channelId: string
  userId?: string
  pattern?: RegExp
  timeoutHandle: NodeJS.Timeout
}
```

`initEventDispatcher(client)` attaches a single persistent `messageCreate` listener at startup. On each incoming message, the dispatcher iterates the Map and resolves the **first** waiter whose filters all match â€” then removes it. If multiple concurrent `wait_for_message` calls are active for the same channel, each receives at most one message; they do not share messages.

The timeout path: `setTimeout` fires after `timeout` seconds, removes the waiter from the Map, and resolves the Promise with `{ timedOut: true, message: null }`.

## Testing Strategy

**`search_messages`:** Inject a mock `client.rest` with a `get` spy. Assert that query params are forwarded correctly and that the response is mapped (outer array flattened, `messages[i][0]` selected, fields extracted).

**`get_messages` pagination:** Assert that `before`/`after` are passed through to `channel.messages.fetch()` when provided, and absent when not provided.

**`wait_for_message`:** `events.ts` exports `_dispatchForTest(msg)` which runs the dispatcher logic synchronously without a real Discord client. Tests:
- Register a waiter via `waitForMessage.handler(...)`, then call `_dispatchForTest(matchingMsg)` â€” expect resolution with message.
- `userId` filter: non-matching userId does not resolve; matching userId does.
- `pattern` filter: non-matching content does not resolve; matching content does.
- Invalid regex in `pattern`: handler throws immediately.
- Timeout: `vi.useFakeTimers()`, advance past timeout, expect `{ timedOut: true, message: null }`.
- Concurrent waiters: two active waiters for the same channel â€” first one gets the message, second remains pending.
